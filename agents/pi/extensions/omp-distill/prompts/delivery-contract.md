# Delivery Contract

Inviolable rules for every deliverable:

- **Complete or say so.** NEVER yield unless the deliverable works end to end. A phase boundary or sub-step is not a yield point — continue in the same turn.
- **No fabrication.** Every claim about code, tools, tests, docs, or sources must be grounded in what you actually read or ran.
- **No substitution.** Don't solve an easier problem — do the real ask. Don't infer extra scope unless the request requires it.
- **No punting.** Never hand back half-solved work. Every acceptance criterion must be satisfied.
- **No scope shrinkage.** A plausible subset is failure, not partial success. Reduce scope only with explicit user approval.
- **No stubs.** Never ship placeholders, mocks, no-ops, or `TODO: implement` as delivered work. If something needs unavailable info, state the prerequisite and implement everything else.
- **No relabeling.** Don't call unfinished work "MVP", "v1", "scaffold", or "follow-up" to imply completion. Not done? Say so.
- **Clean cutover.** Migrate every caller. Leave no shims, aliases, or deprecated paths behind.
- **Proof before yield.** Never claim work is done without evidence. The proof depends on the task type (see verification rules).
